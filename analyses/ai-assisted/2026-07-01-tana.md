# Tana

> **Tana 把"结构化笔记 + Supertags + 知识图谱 + 语音/会议 + AI agents"整合为 agentic 知识工作台；当前 tana.inc 是 agentic meeting platform（"Do work in the meeting"），原 outliner 单独保留为 outliner.tana.inc。**

## 1. 一句话定位

[事实] tana.inc/ 官网首页标题是 *"This is Tana, an agentic meeting platform where AI agents do real work during native video calls"*。副标语 *"Do work in the meeting. Turn team conversations into outcomes, with context that keeps everyone, including your AI agents, in sync."*

[事实] outliner.tana.inc/ 官网首页第一段明确："*This is Tana Outliner, a knowledge-management outliner built on Supertags and nodes. It is a separate product from Tana, the company's agentic meeting platform at https://tana.inc.*"

[事实] outliner.tana.inc 自己的副标语是 *"Your notes, with superpowers. Capture fast, supertag your notes, and watch your information become structured knowledge you can search, filter, and reuse."*

[事实] outliner.tana.inc/company 团队页 — 创始团队三人为：Tarjei M. Vassbotn（CIO / Founder）、Olav S. Kriken（CEO / Founder）、Grim H. Iversen（CAIO / Founder）。

[事实] outliner.tana.inc/blog/the-next-chapter-for-tana/（2026-03-31）宣布 "The next chapter for Tana"，明确把现有 outliner 用户保留在产品线里、新命名是 "Tana Outliner"：

> "To bring the power of Tana to teams, we'll soon be launching a new Tana, designed for real-time collaboration. The current version of Tana will continue as one of two products and will from now on be called **Tana Outliner**."

[判断] **不能再用旧印象"Tana = 笔记软件"理解这家产品和公司**。Tana Inc. 当前有两条产品线：

- **Tana Outliner**（outliner.tana.inc）：面向 power note-taking 用户的 knowledge graph outliner，核心是 **Supertags / nodes / views / AI agents / voice memos / meeting notetaker**，保留 4 年来 community-built 积淀。
- **Tana 新主线**（tana.inc）：面向团队 / 顾问 / VC / 创始人 / 客户成功 / 设计师的 **agentic meeting platform**，核心是 **native video calls + meeting agent + AI agents + Skills + MCP / API + 第三方协作（GitHub / Slack / HubSpot / Linear / Jira / Claude / Cursor / Codex / Lovable）**。

[判断] P30 把 Tana 理解为 **"AI structured knowledge workspace → agentic meeting / work execution platform"** 的转向案例。Supertags + 知识图谱是 Outliner 的产品底盘；在这个底盘基础上，团队把整个产品重新构建为一个 "company brain"，让 AI 在会中、会后、回邮件、写文档、推 GitHub issue 时都使用同一张图谱作为上下文。

**与已有 AI 辅助分析产品的差异点：**

- 与 Notion 的差异：Notion 是 "block + database + AI agent" 集中到文档侧，Tana 是把 Supertag 类型系统、知识图谱、meeting/voice 输入和 AI agents 全做进 outliner / meeting 主线，且 outliner 一条线深度做到几乎"用笔记做软件"。
- 与 Coda 的差异：Coda 是 doc-as-app，build 表格 / 按钮 / Packs；Tana 是 schema-as-outliner + meeting-as-workplace，build 类型系统 / Skills / AI agents。
- 与 Obsidian 的差异：Obsidian 是 local-first Markdown + 双向链接 + 插件生态；Tana 是云端 + 类型系统 + 知识图谱 + 内置 AI agents + meeting agent + MCP，并且已经走出 "knowledge management" 进入 "work execution"。
- 与 Granola / Fireflies / Otter / Read.ai / Fathom 的差异：那些是 meeting note taker；Tana 把 meeting 作为工作起点，AI agents 会在会中完成 deliverable（PRDs / GitHub issues / Jira tickets / HubSpot CRM notes / 客户简报）。
- 与 Replit / Cursor 的差异：Replit / Cursor 给开发者一个 "agentic" IDE；Tana 给 non-engineer 一个 "agentic meeting + workspace"，目标是非工程团队（PM / 顾问 / VC / CS）的会议驱动工作流。

## 4. 产品解决的问题

[判断] Tana（两条线合并）解决的 7 个相互关联的痛点，对应到现有工具的 8 个断裂面：

1. **信息记录和工作执行的分离**：Notion / Coda / Obsidian 把"信息"管得不错，但工作流（filing issue / closing deal / updating CRM / sending follow-up）要切到 GitHub / Linear / HubSpot / Slack。Tana 把工作和信息合在一张图谱上，AI agents 直接执行。
2. **会议纪要 → 工作分发的摩擦**：Granola / Fireflies / Otter / Read.ai / Fathom 录音 + summary 之后，需要人来 follow up；Tana 把 follow up 直接做成 AI agents 在会议内提 proposals，把 done 写回第三方系统。
3. **任务、决策、上下文零散**：Jira / Linear / Slack / GitHub / HubSpot / Docs 各管一段；Tana 用 supertag / type system 把它们拢到一张图，AI chat / agent 通过搜索找 context。
4. **普通笔记缺少结构化类型**：Notion / Evernote 偏向自由 form；Tana supertags 强制结构化，每个 node 都是 typed instance（带字段）。
5. **AI agents 缺上下文图谱**：Mem / ChatGPT / Claude 给的是 chat threads；Tana 提供一张永久图谱作为 agent memory，每次对话都能 pull current state。
6. **会议工具只录音 / 总结，不真正改工作系统**：Granola 等只做 transcription + summary；Tana 把 meeting artifact 直接 draft / issue / file / send。
7. **知识管理工具是"事后整理"，不是"会中执行"**：Obsidian / Notion / Roam / Logseq 都假设用户 daily / weekly 整理；Tana 把整理动作移到 meetings 实时。

[判断] Tana 的产品战略不是 "更好的笔记"，而是 "**the real problem isn't note-taking, it's how work itself is structured**"（出自 the-next-chapter-for-tana blog）。这是它和 Notion / Coda / Obsidian 最大的产品哲学差异。

## 6. 核心用户路径

### 6.1 Tana Outliner 核心路径

```
注册 → 创建 workspace
      → 新建 daily note（自动激活 today view）
      → 输入 bullet / node（outline editor）
      → 添加 supertag（#task / #project / #meeting / #contact）
      → 定义字段（status / owner / due date / link）
      → saved view 聚合（按 owner / due / project 自动聚合）
      → AI chat with notes（@Tana AI 用图谱回答）
      → voice memo 录音（桌面 / 移动，自动转写 + 打 supertag）
      → 生成 publish 共享（密码保护 / 公开链接 / 全图谱交互）
      → template 安装（从 templates gallery 安装可执行 workflows）
```

**路径分析：**

- **步骤数**：8 步完成基本使用；11-12 步完成 advanced 场景。
- **关键决策点**：选择 supertag 类型（task / project / meeting / custom）决定后续 saved view / automation 的可能性。
- **流失点**：
  - 第一次接触 supertags / fields：门槛较高，"它是什么 / 它和文件夹有什么区别" — 这是新手最大流失点。
  - AI credits 限制：Free 500 / Plus 2000 / Pro 5000 — 中度使用可能超支。

### 6.2 Tana 主线（agentic meeting platform）核心路径

```
注册 / SSO
   → 接入 Google Calendar / Outlook（同步今天会议）
   → 接入 GitHub / Slack / HubSpot / Linear / Jira
   → 创建一个 space（产品 / 团队 / 客户 / 投资项目）
   → 创建第一个 AI agent：
       · 配 memory（团队 OKR + 客户背景）
       · 配 Skill（"Pull last week's milestones" / "Draft client brief" / "Update CRM"）
       · 配 integrations（GitHub / Slack / HubSpot）
   → 加入 video call（native 或 Zoom/Teams/Meet botless）
   → AI agent 在会中捕获对话 → 列出 Decisions / Owners / Next steps（写入 Tana）
   → 通过 proposals 提交到第三方系统：
       · GitHub issue filed
       · Slack message sent
       · HubSpot deal updated
       · Linear issue created
       · Jira bug filed
   → 下次会议 — AI 已经 pull 历次会议 context，无需重新说明背景
```

**路径分析：**

- **步骤数**：6 步注册 + 5 步日常使用，与会后直接 artifact delivery。
- **关键决策点**：配 AI agent 是 4 个维度 — memory / Skill / integrations / reviews，这是 Tana 主线用户体验的核心摩擦。
- **可能的流失点**：
  - "So much setup" — Tana 自己承认新主线也需要学习 Skills / integrations，Next Chapter blog 也写到这一点。
  - "Proposals review" 信任度 — AI 写回第三方前必须人工 review，这是摩擦但也是合规。
  - 企业 IT onboarding：SOC2 ETA Q3 2026 / HIPAA ETA Q3 2026 — 当前合规范围部分 incomplete。

### 6.3 Path 对比

| 维度 | Outliner 路径 | 主线路径 |
|------|--------------|---------|
| 起点 | 个人 daily notes | 团队 space |
| 终点 | 保存 / 分享 / 模板安装 | artifact delivery to 3rd party |
| 主心智 | "build a type system over time" | "configure AI once, run every meeting" |
| Learning curve | 陡（supertags / commands / views） | 中（需要理解 Skills + integrations） |
| Day-1 value | 中（empty workspace is empty） | 高（calendar 自动 populate） |

[判断] 两个路径其实指向同一张图谱（graph）。Outliner 老用户的 supertag 结构在新主线的 "types" 上继续使用，新主线的 meetings → decisions 在 Outliner 的 saved view 里能看到。两条线的"骨架"一致（graph + types + AI agents），只是入口不同。

## 12. 主要优点

[判断] Tana 的 7 个主要优点：

1. **双产品线抓住所有用户场景**：个人 PKM（Outliner）+ 团队工作流（main）— 用户可以从 Outliner 升级到主线，也可以直接进入主线；不像 Notion / Coda 必须选个入口。

5. **Native video calls + botless meeting capture**：既支持 native video calls，也支持 botless（"no moving the call or asking anyone to switch"）从 Zoom / Teams / Meet 抓取。
6. **MCP / API first 与 LLM agnostic**：与 Anthropic MCP 生态对齐，用户可切换 model；Tana 不锁死到单一 LLM。
7. **Community-built with 160K waitlist + 24K+ Tanarians**：Tana outliner 用户参与从产品开发到模板建设到 live sessions，community 健康度可与 Obsidian / Linear / Figma 类比。

## 13. 主要问题

[判断] Tana 的 7 个主要问题：

1. **学习曲线陡峭**：Supertags / fields / views / command nodes / supertag instances / live searches / AI command nodes — Outliner 老用户可能能玩起来，但切换新用户会遭遇同样 friction that Tana Outliner 一开始就被批评的 "It's powerful, but hard to get into"。tana.inc/company 也写到 Outliner 团队很多人 = power users，新主线用户群 ≠ Outliner 老用户群。

3. **private company, $25M 全部融资**：相比 Anthropic / OpenAI / Cursor / Replit，Tana 资金规模小；outliner 老用户担心公司能否长期 sustain / 收购。
4. **Outliner / 主线的未来路径不清**：Tana "两线并行" — Outliner 保留 + 主线新做。这意味着 Outliner 老用户的"建立多年的 type system"是否会迁移到主线，或者主线与 Outliner 是否会并行多年而稀释团队资源？Next Chapter blog 直接说 "There will be a period when the new Tana does not yet do everything Tana Outliner can do. That gap will be real."
5. **AI credits 限制 outliner**：Outliner Free 不能 top-up credits，Plus 2000 / Pro 5000 — 中度使用 / power user（学生做大量 lecture transcription）容易触顶。
6. **tana.inc/learn 还重定向到 outliner.tana.inc/learn**：说明主线 docs 还没完全独立站点化，对企业 IT 来说成熟度可能受限。
7. **Personal user vs team user 切换成本**：Outliner 老用户在 supertag 系统沉淀多年，迁移到主线不可避免要重新建类型 / saved view / agents — Tana 必须在"保留 Outliner 老体验"和"让主线变简单"之间谨慎前进。
8. **Meeting agent "会中执行" 的真实可靠性未亲验**：botless meeting capture / Skills 自动产出 PRD / GitHub issue / HubSpot CRM update 等会在会议内直接 deliverable，但 tana.inc 官方页面只是产品 demo 截图 / 客户 quote，没有第三方独立 verified 评测（如 TechCrunch 实测 / WIRED 实测 / Forrester analyst 评测）。AI 自动写入工作系统可能需要强权限 + 治理 + 失败 fallback — 这些可靠性 / 安全治理 / 失败兜底机制目前主要靠 "Proposals review" 兜底，对错误 transcribe / hallucination 的失败模式没有公开 verdict。

## 15. 对我自己项目的启发

[判断] Tana 对 Product-Analysis 的启发是 6 条：

1. **"The real problem isn't note-taking, it's how work itself is structured"**：这是 Tana 的核心反思。Product-Analysis 的核心也不是"分析了几篇产品"，而是"如何把分析变成可重用的资产管理" — 每篇分析应该既是记录又是 structured asset（类似 supertag / view）。
2. **Supertags + Knowledge Graph 双层结构**：可以在 analyses/ 下引入 `ai-structured-tag:` 类型系统 — 每篇 AI 分析是一个 `analysis` supertag，每个 Tana / Coda / Obsidian 等产品是 `product` supertag，每个 startup / 公司是 `company` supertag。这样 P30 的 Tana 既能 link 到 Notion / Coda（产品），也能 link 到 Anthropic / Datadog / Dropbox（公司 / 投资人）。
3. **Graph 优于 Folder**：Tana Outliner 的核心 insight 是 graph 比 folder 强大（notes 之间用 @ 链接）。Product-Analysis 的 `analyses/` 目录可以引入内部交叉引用 `[[product-name]]`，让 13 个 product 分析互相 link，形成 view。
4. **AI agents + Skills 框架的迁移**：把 Superpowers 风格（SOUL.md 行为准则）的 agents 概念应用到 Product-Analysis — 每个分析类（AI-assisted / 人工 / 旧）是一个 Skill，validator 是 Agent。
5. **MVP 应该 strip "anything awesome" to "one thing"**：Tana Outliner 的 friction 是 "太 powerful 太复杂"。Product-Analysis 应该继续 keep it simple — 13 篇 AI assisted 分析 + 9 篇 legacy + 1 product hunt 复盘 — 这是 scope discipline。
6. **MCP / API first 是 AI 工具的标准**：未来如果 Product-Analysis 引入 MCP server（基于文件系统），Claude Code / Cursor / Codex 可以 query Product-Analysis — 这是 Tana 2026-01-30 发布 Local API + MCP 给出的方向。
7. **Source-first + Source verification 是 AI 辅助分析的稳定器**：Tana 分析中虽然很多第三方媒体 unavailable（Cloudflare 403 / paywall），但 tana.inc/company + 官方 blog 自报的事实仍然构成 main source — 这是 source-first + source verification 双重保护：主体产品机制由官方源 verified，高风险事实（融资 / 估值 / 收入）则在可信度分级表中明确 partial。这是 Product-Analysis 的 source-quality-checklist v1.0 与 source-hardening 方法在新分析中的成功复用 — 确保 每篇 AI 辅助分析在缺源情况下也保持诚实而非假装 verified。

## 16. 今日复盘

[判断] 今日复盘 3 条：

1. **Tana 是 "structured outliner → agentic meeting/work platform" 的转向案例**。Tana Inc. 当前有两条独立产品线：tana.inc 是 agentic meeting platform，outliner.tana.inc 是 Tana Outliner。两个站点、一套公司、双向共享"graph + supAI agents"哲学。P30 不能再用旧印象"Tana = 笔记软件"理解，必须区分主战场。

2. **Tana 的核心产品哲学是 "graph is the context, conversations are your prompts"**。这与 Notion / Coda / Obsidian 都是"文件 / 文档 / 笔记侧"不同 — Tana 把"会议"与"知识图谱"绑定，AI 不再 from blank page 开始。这是 AI-native workspace 的下一阶段 — 让 agents 在团队沟通时已经在 context 中工作（"less setup, more happening automatically"）。

3. **双线并行 = 高风险 vs 高价值的战略赌博**。Outliner 用户担心类型系统迁移，主线用户需要从零学习 Skills / integrations / reviews。Tana 现在的策略是 "如果它 works for you, 你可以继续用" — Outliner 独立 maintain，新主线持续 build。这是 Coda 被 Grammarly 收购、Notion 持续扩张、Obsidian 稳定 PKM 之外的第四条路 — Tana 想做"agentic workspace"，赌的是"未来所有 work 都将是 AI-mediated"。
