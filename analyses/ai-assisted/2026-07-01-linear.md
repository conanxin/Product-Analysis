# Linear

> Linear 不只是 issue tracking，而是一个**为高速产品研发团队设计的工作流操作系统**——从首次进入产品的速度感，到与 GitHub / Slack / Figma 的深度集成，每一步都把"用得起来"放在"功能齐全"之前。

---

## 1. 一句话定位

Linear 是一款面向**现代软件团队**的 issue tracking、roadmap、project planning 与 product development workflow 工具。它不是传统意义的"任务管理工具"（不像 Jira/Asana/Trello 那种通用项目管理），也不是"文档协作工具"（不像 Notion/Confluence 那种知识管理），而是**为产品研发节奏本身设计**的工作流系统——强调速度、清晰度和工程师使用意愿。

## 2. 目标用户

[事实] 从 Linear 官网和 TechCrunch 报道观察，Linear 的核心用户可分成 5 类：

| 用户类型 | 典型特征 | 用 Linear 解决什么问题 |
|----------|----------|-----------------------|
| **创业公司产品研发团队** | 5-50 人，迭代速度是核心竞争力 | 替代 Jira/Asana；用统一工具跟踪 issue/cycle/project |
| **工程师** | 工具栈复杂（Git/GitHub/Slack），讨厌臃肿工具 | 体验顺畅的 issue 工具；键盘驱动；与开发工具链整合 |
| **产品经理** | 需在 issue / project / roadmap / cycle 之间协调 | 单一界面同时管理细节和全局 |
| **设计师** | 经常与工程师协作 | 通过 Figma 集成直接关联设计文件 |
| **工程管理者 / 团队 lead** | 需在团队/项目/产品层做规划 | Cycles + Roadmaps + Projects 三层视图 |

[判断] 这 5 类用户中，**最关键的是工程师群体**——Linear 的产品设计明显以"让工程师愿意打开"为最高优先级（页面切换速度、键盘快捷键、GitHub 集成）。这是它区别于 Jira 的根本定位：Jira 假设用户是 PM，Linear 假设用户是工程师。

## 3. 核心场景

[事实+判断] Linear 在以下 6 个核心场景表现最强：

| 场景 | 描述 | 体现的核心价值 |
|------|------|----------------|
| **Issue tracking** | 跟踪 bug、feature、task、improvement | 极快的创建/编辑/筛选体验；键盘优先 |
| **Cycle / Sprint** | 周期性迭代规划（2 周一个 cycle） | Cycles 模块；自动 carry over |
| **Project planning** | 跨 cycle 的主题项目 | Projects 模块；milestone；进度追踪 |
| **Roadmap** | 季度/年度规划视图 | Roadmaps 模块；按时间轴展开 |
| **Triage** | 收到 issue 后快速分类和优先级 | Inbox/Triage 模块；批量处理 |
| **Integration workflows** | 与 GitHub/Slack/Figma/Sentry/Intercom 同步 | Issue 与代码 PR/设计/客服记录直接关联 |

[判断] Linear 的"杀手场景"是**研发节奏管理**——从 cycle 内执行到 cycle 间规划，再到季度 roadmap，**三层时间尺度**在同一工具内统一管理。这是它能替代 Jira 的根本原因：Jira 是 issue 数据库，Linear 是产品研发操作系统。

## 4. 产品解决的问题

[事实] 从用户痛点出发，Linear 同时解决 5 个问题：

1. **Jira 太慢** — 页面加载、issue 切换、评论输入都有明显延迟
2. **Asana / Trello 偏轻量** — 不支持复杂研发场景（cycle、triage、roadmap）
3. **Notion 太自由** — 灵活但缺乏流程；issue 容易散落
4. **GitHub Issues 太简单** — 工程师喜欢，但 PM 觉得不够结构化
5. **工具割裂** — issue 在 Jira，需求在 Notion，roadmap 在 Sheets，三者不同步

[判断] 这 5 个问题中，**"工程师不愿用 Jira"**是 Linear 的根本命题。Linear 团队（前 Designers / Engineers from Apple/BMW/Coinbase）的设计哲学是把 Linear 做成"工程师也愿意每天打开的工具"，这与 Jira 把工程师视为"被动接受 PM 安排"的思路是根本对立。

## 5. 首页与第一印象

[事实] 来自 linear.app 官网观察：

- **首页叙事结构**：开屏 hero（"Linear is a purpose-built tool for planning and building products"）→ 价值主张 → 截图展示 → 客户 logo → 集成列表 → pricing
- **视觉语言**：极简、黑白为主、品牌色紫蓝、抽象几何背景
- **动效**：极少；强调"快"和"清晰"
- **品牌一致性**：method 页面、docs 页面、blog 页面保持同样视觉系统

[判断] 首页设计哲学是"少即是多"——没有促销、博客推送、复杂导航，**只是**清晰地告诉用户"我们做什么 + 我们做得快"。这种克制本身就是品牌信号。

> 注：本节只观察官网营销页，**未亲验产品本身**。

## 6. 核心用户路径

[事实+判断] 基于官方 docs 描述推断的核心用户路径：

```
注册账号 → 创建 workspace → 邀请团队 → 创建 team
  ↓
第一次创建 issue（C=快捷键）
  ↓
添加 project / cycle / label / priority
  ↓
收到 GitHub/Slack 通知 → Triage（Inbox）查看
  ↓
排入 cycle → 关联 GitHub PR → 自动化状态更新
  ↓
cycle 结束 → 查看 progress → 启动下一个 cycle
  ↓
季度复盘 → 用 Roadmap 视图对齐产品方向
  ↓
发布 → 用 changelog 通知用户
```

[判断] 这条路径的关键创新是**与开发工具链的深度整合**——创建 issue 后，几乎所有状态变化（PR 合并、Slack 提及、Figma 更新）都能自动同步回 Linear。**用户不必手动维护状态，工具自动收口**。

## 7. 信息架构

[事实] 基于 linear.app/docs 观察，Linear 的 IA 围绕"工作流对象 + 时间维度"展开：

| 对象 | 作用 | 所属层级 | 设计意义 |
|------|------|----------|----------|
| **Workspace** | 整个组织 | 最上层 | 跨团队协作容器 |
| **Team** | 单一产品/部门 | workspace 下 | 每个 team 独立设置 |
| **Issue** | 单一任务 | team 内最细粒度 | 跟踪具体工作 |
| **Project** | 主题项目（跨 cycle） | team 内 | 把 issue 按主题组织 |
| **Cycle** | 周期性迭代（2 周） | team 内 | 把 issue 按时间组织 |
| **Roadmap** | 季度/年度规划 | workspace 视图 | 把 project 按时间组织 |
| **View** | 自定义筛选 | team 内 | 动态聚合 issue（My Issues / Triage / All） |
| **Inbox** | 通知/待处理 | team 内 | 集中处理新 issue 和 mention |
| **Label** | 横向分类 | team 内 | 跨 cycle 跨 project 的属性 |
| **Status / Priority** | 标准字段 | team 内 | 统一语言 |
| **Integration** | 外部工具接入 | team 内 | GitHub/Slack/Figma 等 |

[判断] 这种 IA 设计的关键是**三层时间尺度的统一**：
- **短**（issue）→ **中**（cycle/project）→ **长**（roadmap）
- 在同一 UI 内能无缝切换视角
- 这是它对 Jira"项目 + 子任务 + sprint + epic"多层抽象的重新组织

## 8. 交互与视觉设计

[判断] Linear 的交互设计有 5 个值得注意的细节（基于官网/截图）：

| 交互点 | 描述 | 评价 |
|--------|------|------|
| **键盘优先** | C 创建 issue、Cmd+K 搜索、所有操作有快捷键 | ✅ 工程师友好 |
| **即时反馈** | 编辑/保存几乎无延迟；UI 状态切换动效简洁 | ✅ 速度感是产品核心 |
| **左中右三栏布局** | 导航 + 列表 + 详情；与 macOS 原生应用风格统一 | ✅ 跨平台一致 |
| **同步引擎** | 编辑同步实时；本地优先；可离线操作 | ✅ 技术感 |
| **键盘图标与渐变** | 视觉系统克制；不堆特效 | ✅ 让人专注工作 |

[判断] Linear 的视觉风格与 Linear Method 哲学一致——**"built, not designed"**——不强调"漂亮的 UI"，强调"顺手的工具"。这与 Notion 的"自由灵活"形成对比。

[判断] Linear **最被低估的增长机制是 Linear Method 文档**——这不是 marketing collateral，而是产品哲学的公开表达。在"工具堆里"，Linear 用方法论（而非功能）建立差异化，吸引认同其价值观的团队。

## 10. 商业模式

[事实] 来自 linear.app/pricing 观察：

| 计划 | 描述 | 目标 |
|------|------|------|
| **Free** | 基础功能，有用户数和成员数限制 | 个人 / 小团队拉新 |
| **Standard** | $8/月 per user | 主力小型团队 |
| **Plus** | $14/月 per user | 进阶需求（time tracking, customer requests） |
| **Enterprise** | 联系销售 | 大客户定制 |

[事实+判断] 2019 年获 $4.2M A 轮（Sequoia 领投）；2025-06 获 $82M 融资，估值 $1.25B（TechCrunch 2025 报道）。**没有广告，**完全订阅制。

[判断] 商业模式的核心矛盾是：**工程师团队在产品研发工具上付钱意愿其实不高**（Jira 价格相对便宜），Linear 必须证明"节省的时间"和"更好的体验"值 $14/月/用户。对高速迭代团队（特别是工程师是核心生产力的公司）这个定价可行，但对传统企业销售流程是挑战。

## 11. 竞品与替代方案

[事实+判断] 竞品对比：

| 产品 | 核心优势 | 核心弱点 | Linear 的差异点 |
|------|----------|----------|----------------|
| **Jira** | 市场份额最大、生态最全 | 慢、UI 复杂、工程师反感 | Linear 速度快 10x；UX 优先 |
| **GitHub Issues** | 与代码深度集成 | 不支持 roadmap、project、cycle | Linear 有 cycle / project / roadmap |
| **Asana** | 通用、轻量 | 不针对研发场景 | Linear 为研发场景定制 |
| **Notion** | 灵活、能做一切 | 缺乏流程；状态靠手动 | Linear 自动同步；状态自动 |
| **Trello** | 简单看板 | 不够结构化 | Linear 同样简单但更结构化 |
| **Shortcut (Clubhouse)** | 早期产品开发定位 | 品牌不如 Linear | Linear 后来居上 |

[判断] 真正的竞争对手不是 Asana 或 Trello，而是**"工程师团队的内部共识"**——很多小团队用 GitHub Issues + Notion + Slack 凑合。Linear 要让团队相信"统一工具"是值得付费的。2025 年 $1.25B 估值说明这个故事已经被市场接受。

## 12. 主要优点

1. **速度** — 整个产品的核心，所有交互以毫秒计
2. **键盘优先** — 让工程师能用 vim 思路操作工具
3. **开发工具链整合** — GitHub/Slack/Figma/Sentry 深度同步
4. **三层时间尺度统一** — issue / cycle / roadmap 在同一 UI
5. **清晰的信息架构** — 复杂功能但不混乱
6. **方法论驱动** — Linear Method 让产品有灵魂
7. **公开 Changelog** — 透明的产品演进

## 13. 主要问题

1. **对非技术团队不友好** — PM/设计/运营上手成本相对高
2. **个性化受限** — 不如 Notion 灵活，自定义字段有限
3. **小团队可能觉得贵** — $14/月/用户，对 5 人团队就是 $840/年
4. **缺乏文档 / 知识管理** — 不像 Notion / Confluence 能写文档
5. **国际化** — 主要是英文；中文支持有限
6. **大企业销售** — 销售流程相对轻；企业级市场渗透有限
7. **2025 估值 $1.25B 后续增长压力** — 已被称为 "Atlassian rival"，但 Jira 仍主导
8. **生态较新** — 集成 / 插件生态不如 Jira 成熟

## 14. 如果我来重做

假设在中文语境下做一个"类 Linear"的产品，**MVP 范围不能复制 Linear 的全部**——因为 Linear 的成功依赖与 GitHub/Slack/Figma 等英文生态的深度整合。

**MVP 建议方向**：

- **目标用户**：中文互联网的产品研发团队（特别是 SaaS 公司）
- **差异化定位**：与飞书项目、Notion Chinese、Tower 区分
- **特殊问题**：
  - 中文互联网的代码托管分散（GitHub/Gitee/自建 GitLab）
  - IM 生态分散（飞书/钉钉/企业微信）
  - "Issue" 这个概念在中文化时是"工单"还是"事项"需要本地化

**建议切入方向**：

- 与 **飞书项目 / Lark 生态** 深度整合（不与 Notion 直接竞争）
- 在 **创业公司** 这块抢占，避开大企业市场
- 强项放在 **AI 加持的自动 triage** 与**自动状态同步**——这是 Linear 还没完全做到的
- **价格**：中文市场对价格敏感，建议 Free + 中端 $5/月/用户 + 企业版联系

[判断] 最深的一层启发：**Linear 的成功不在功能多，而在"有立场"**——它不只是"另一个 issue tracker"，而是一种对"工程师如何使用工具"的态度表达。Product-Analysis 也需要"有立场"——不只是"产品分析文章仓库"，而是一种"如何从产品中学习"的方法论。

## 16. 今日复盘

Linear 表面上是"另一个 issue tracking 工具"，本质上是一套**为产品研发节奏设计的工作流操作系统**。它的成功不是"功能齐全"，而是"有立场"——对工程师友好、对速度执着、对整合开放、对方法论透明。

对 Product-Analysis 项目的最大启发是：**仓库也可以有立场**。本仓库 P1-P4 已经建立了模板、框架、来源质量门禁；P5 继续；未来要持续回答"我们用什么方法分析产品、为什么这样分析"。
