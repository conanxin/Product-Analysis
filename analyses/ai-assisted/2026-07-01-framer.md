# Framer

## 1. 一句话定位

Framer 不是普通 website builder，也不是 Figma-to-code 工具的简单延伸。Framer 是一个以**设计画布（Canvas）为核心**、深度集成 **CMS、AI agents、发布、SEO、模板生态**的 **AI website builder / no-code web publishing platform**（[判断] 分析性概括，官网原文 "AI website builder for professional sites" verified），目标用户是设计师、独立开发者和创业团队（[判断]），核心价值是让"设计"和"上线"之间不再需要工程师介入（[判断] 分析性概括，非官方口号）。

官网口径：**"Framer: AI website builder for professional sites"**（framer.com/ 页面 verified）。

## 2. 目标用户

Framer 的目标用户不是普通消费者建站市场，而是有设计感和品牌要求的 B2B / Creator / Startup 受众：

- **设计师**：希望把设计稿直接变成可发布网站，不需要工程师
- **独立开发者 / Indie Hackers**：快速搭建 landing page / 产品官网，低代码甚至零代码
- **初创公司**：用模板快速上线，配合 AI agent 快速迭代 marketing site
- **市场团队**：通过 on-page editing 和 CMS 自主更新内容，减少对工程师的依赖
- **创意工作室 / Agency**：用模板生态快速交付高质量客户网站
- **模板创作者**：通过 marketplace 生态变现设计资产
- **非技术内容编辑**：通过 CMS + AI agent 管理内容，不需要理解代码
- **企业网站团队**：通过 Enterprise 套餐获得 SSO、SCIM、权限控制

## 3. 核心场景

Framer 覆盖以下核心使用场景：

| 场景 | 描述 | 对应 Framer 能力 |
|------|------|----------------|
| Landing page / Marketing site | 快速搭建高质量产品介绍页 | 模板 + AI agent 生成 + 视觉质量高 |
| Portfolio / Creator site | 设计师 / 创作者个人展示站 | 模板生态 + 视觉优先 + 移动端优化 |
| Startup website | 创业公司快速上线，配合融资/招聘 | AI agent 快速生成 + CMS 内容更新 + SEO |
| CMS-driven blog / content site | 需要持续更新内容的知识库/新闻站 | CMS collections + AI agent 内容管理 |
| Template-based site creation | 基于模板快速建站 | 大量模板 + Marketplace |
| Design-to-publish workflow | 设计稿直接发布，不需要开发 | Canvas → Publish 直接链路 |

## 4. 产品解决的问题

Framer 的存在源于以下行业痛点：

**设计稿与真实网站的割裂**：传统工作流是 Figma 设计 → 交付工程师 → 工程师用代码实现，Framer 把这个链路压缩为"设计即上线"。

**Webflow 功能强但学习成本高**：Webflow 是专业级的 website builder，但学习曲线陡峭，Framer 用更现代的 UX 和 AI 降低门槛。

**WordPress 生态庞大但现代设计体验弱**：WordPress 适合复杂 CMS，但模板视觉质量参差不齐，技术债务累积快。

**营销团队改文案慢**：传统建站工具中，营销团队改一个字都需要工程师介入，Framer 的 on-page editing 和 CMS agent 让非技术团队自主运营。

**CMS 与设计画布割裂**：很多建站工具的 CMS 和设计编辑器是分开的，Framer 把 CMS collections 直接连接到 Canvas，内容和设计保持同步。

**AI 生成页面后难以进入可编辑生产流程**：很多 AI 建站工具生成的页面是不可编辑的图片或 HTML，Framer 的 AI agent 生成的是可直接在 Canvas 内编辑的组件，保持设计可控性。

## 8. 交互与视觉设计

**设计画布与网站发布合一**：Framer 的核心交互模型是 **Canvas = Published site**。设计师在 Canvas 上的操作直接映射为网站输出，不需要额外的"预览 → 导出 → 上传"流程。Canvas 支持响应式 breakpoints，在 desktop / tablet / mobile 视图间切换编辑。

**Layer / Component Model**：Framer 继承了 Figma 式的 layer 面板，设计师可以管理每个元素的样式、位置、动画。Components（组件）是可复用单元，支持 variants（变体），让设计师能构建系统级设计资产。

**CMS Table 与 Canvas 连接**：CMS collections 的每个 field 可以直接绑定到 Canvas 内的文本、图片、链接。这意味着当 CMS 内容更新时，页面自动刷新，无需重新发布。CMS agent 可以从 Slack / terminal / GitHub PR 触发，自动填充或更新 CMS 内容。

**On-page Editing**：发布后的网站，非技术用户可以直接在页面上双击编辑文本内容，修改直接写回 CMS collections，保持内容和设计的一致性。

**AI agents 的 Canvas 嵌入**：
- **Design agent** [事实 / 官方 demo]：在 Canvas 内接收 prompt，生成 layout sections、修改 typography/spacing/color，生成结果直接在 Canvas 内可见可编辑
- **CMS agent** [事实 / 官方 demo]：管理 collections（创建 field、填充 CSV、批量更新内容），连接 CMS 和 Canvas
- **Code agent** [事实 / 官方 demo]：在 Canvas 内生成 custom effects / 复杂交互代码片段（ImageWheel code agent demo verified）
- **真实可用性 / 质量 / 效果好坏 / 主导增长程度** [判断]：以上为官方 demo 描述，真实用户体验、refinement 次数、credit 消耗效率、采用率等没有独立数据

**Code Components / Overrides**：对于开发者，Framer 提供 Code Components（自定义 React 组件）和 Overrides（修改现有组件行为）。通过 Framer Server API 可以实现更复杂的 backend 集成。

**为什么 Framer 比传统 website builder 更接近设计师心智**：传统建站工具（Wix / Squarespace / Webflow）使用的是"区块拖拽"模型，而 Framer 使用的是"设计编辑器"模型，设计师在 Canvas 上的操作逻辑与 Figma / Sketch 一致，学习成本从"学新工具"降为"熟悉快捷键"。

## 10. 商业模式

**Pricing 层级**（framer.com/pricing/ verified）：

| 套餐 | 月费（年付） | 核心限制 | AI Credits |
|------|------------|---------|-----------|
| Free | $0 | 500 credits，Framer subdomain，1 site，1 page，30 种布局变体 | 500 one-time |
| Basic | $10 | 1,000 credits/mo，自定义域名，2 CMS collections，50 GB 带宽，SEO，内置 SEO 工具 | 1,000/mo |
| Pro | $30 | 3,000 credits/mo，10 CMS collections，100 GB 带宽，staging 环境，site redirects，advanced hosting（add-on），A/B testing（add-on） | 3,000/mo |
| Enterprise | Custom | 自定义 limits，企业级安全，Uptime guarantee，SCIM SSO，专属支持 | Custom |

**Site-based Pricing**：Framer 按 site 数量和站点资源计费，不是按页面或流量计费。Basic 支持最多 30 个自定义页面，Pro 最多 150 个（超出后 $20/100 pages）。

**AI Credits**：AI agents 和其他 AI 功能消耗 credits。Design agent 生成一次 layout 或修改一次 section 消耗固定 credits。Credits 按月计算，未用完不累计。External agents 连接不消耗 Framer credits（由外部 AI provider 计费）。

**Templates Marketplace**：模板创作者可以在 marketplace 发布模板并获得收入分成。Framer 从中抽成，形成双边市场。

**AI Features 作为付费驱动**：AI agent 功能是 Pro 以上套餐的核心卖点，Basic 只有 1,000 credits/月，Pro 有 3,000 credits/月。Enterprise 的自定义 limits 包含 unlimited editors + 自定义 AI credits 套餐。

## 12. 主要优点

1. **设计到发布链路极短**：Canvas → Publish，没有工程交付环节，设计师自主完成全流程
2. **面向设计师的交互模型**：Layer / Component / Variants / Overrides 体系与 Figma 一脉相承，设计师零学习成本
3. **视觉质量高**：模板生态展示的案例（Microsoft / Zapier / Loom）证明 Framer 能承载高端品牌网站
4. **AI agents 的 Canvas-native 集成**：AI 生成结果直接在 Canvas 内可编辑，不破坏设计控制权，这是与其他 AI 建站工具的本质区别
5. **CMS 与 Canvas 实时连接**：CMS collections 绑定 Canvas 元素，内容更新自动反映在页面上，不需要重新发布
6. **On-page editing 赋能非技术团队**：内容编辑直接在页面上进行，减少了对工程师的依赖
7. **模板生态 + Marketplace**：快速启动 + 设计资产变现，形成正向循环
8. **外部 AI agents 开放整合**：Claude Code / Codex / Cursor 可以连接到 Framer，作为 AI agent workflow 的中枢
9. **多语言 / SEO / Analytics 内置**：不需要第三方插件，降低技术复杂度
10. **AI credit 透明计费**：按用量计费，Free tier 可试用，降低用户尝试门槛

## 13. 主要问题

1. **复杂网站和应用型产品能力边界**：Framer 适合 marketing site / portfolio / landing page，但复杂 Web app（需要 database transactions / 复杂用户系统）能力不足
2. **与 Webflow / WordPress 生态相比仍有差距**：Webflow 的 CMS 和 logic 体系更成熟，WordPress 有数万个 plugins，Framer 的生态深度还需要时间积累
3. **设计师友好不等于所有营销团队都易用**：虽然比 Webflow 简单，但 CMS collections 配置、SEO 设置、AI agent prompt 仍需要一定学习曲线
4. **AI agent 生成质量依赖 prompt 能力**：Design agent 生成的 layout 质量高度依赖用户描述的清晰度，非设计师用户可能难以获得理想结果
5. **Pricing / site-based 限制可能造成摩擦**：随着站点数量增长，费用也线性增长，Agency 场景下成本可能高于预期
6. **过度追求视觉可能影响性能 / 可维护性**：Framer 生成的页面可能包含过度复杂的 CSS/JS，性能优化依赖 Framer 基础设施
7. **企业级合规、权限和迁移成本**：Enterprise 套餐提供 SSO/SCIM，但大型企业的合规需求（GDPR / SOC 2）需要额外验证；迁移出 Framer（到 WordPress/Webflow）需要重新设计
8. **AI credits 消耗可能过快**：一次完整的 AI 生成流程（多个 sections 调整 + 多次 refinement）可能在几分钟内消耗数百 credits，Pro 3,000/mo 可能不够高频用户
9. **与 Figma 的定位重叠可能导致内部竞争**：Figma 也在推 Figma Sites，Framer 和 Figma 的差异化是否持续是个战略问题

## 14. 如果我来重做

**核心判断**：在中国语境下做一个"Framer 类"产品，不应该直接模仿 Framer 的全部功能，而应该先做垂直场景。

**中文网站建设痛点**：
- 微信生态（小程序 + 公众号 + 官网）需要打通，但三者设计语言不统一
- 飞书文档 / Notion 作为内容中台，需要与网站同步
- 小红书 / B站成为主要品牌触点，官网需要与社交媒体联动
- 企业官网 + 产品文档 + 定价页面需要统一维护，但团队通常没有专职前端
- AI 建站的价值：让产品经理 / 市场运营能自主更新官网，不需要工程师介入

**应该先做完整 Webflow / Framer 吗**：不应该。完整 Webflow 级别的功能需要 3-5 年的产品积累。先做垂直场景，验证用户需求，再横向扩展。

**MVP 建议**：面向 AI 原生个人项目和小团队的产品官网生成器

核心功能：
1. **Prompt → Landing Page**：输入产品描述 / 关键词 → AI 生成结构化 landing page（Hero / Features / Pricing / CTA）
2. **产品信息结构化采集**：引导用户输入：产品名 / 一句话介绍 / 核心功能（3-5 条）/ 团队背景 / 截图/demo 链接 → 生成完整官网内容
3. **Screenshot / Demo embed**：支持嵌入产品截图、Loom 视频、GitHub Stars 等社交证明元素
4. **Pricing / Changelog / Docs 页面**：自动生成标准页面，减少手动维护
5. **SEO metadata**：自动生成 meta title / description / OG image
6. **多语言**：基于 LLM 翻译，一键生成中英双语官网
7. **Agent 自动更新文案**：连接微信客服 / 飞书机器人，让非技术成员能通过对话更新网站内容
8. **GitHub / Product-Analysis 生成 landing page**：给定 GitHub repo URL，自动抓取 README、stars、contributors，生成产品官网

**如何避免成为"又一个模板建站工具"**：核心差异化是 AI agent 工作流。传统模板建站：选模板 → 改文字 → 发布。AI 建站：描述需求 → AI 生成 → 人工审核 → 发布 → AI 持续优化。关键是"人机协作"而不是"AI 全自动"。

## 15. 对我自己项目的启发

结合 Product-Analysis 本项目，Framer 的产品逻辑给了我以下启发：

1. **Product-Analysis 可以从 Markdown 仓库变成可发布站点**：每篇产品分析（Perplexity / Linear / Raycast / Cursor / Figma / Framer）可以自动生成独立的可浏览页面，不只是 GitHub repo 里的 Markdown

2. **每篇产品分析可以自动生成 landing page**：给定产品名 + 核心洞察 → AI 自动生成一页展示页面，带 Sources 可信度评级 / review status / 核心结论

3. **Sources / credibility / review status 可以成为页面模块**：每个产品的 source_url_verification_status（verified / partial）可以变成页面的"可信度徽章"，让读者一眼判断分析质量

4. **Reports 可以变成 Changelog**：每篇分析对应一个 P# 报告，这些报告可以变成站点的 Changelog 页面，展示产品研究的时间线

5. **README 索引可以变成 CMS collection**：README 的 AI 辅助分析索引表格（日期 / 产品 / 状态 / 来源状态）本质上就是一个结构化的 CMS，可以迁移到 Framer CMS 做可视化管理和 AI 自动更新

6. **AI agent 可以自动更新文章页和索引**：当新的 AI 产品分析完成后，AI agent 自动抓取关键数据（融资 / 用户数 / 核心功能）并更新对应页面，不需要手动同步

7. **未来可做"产品研究站点生成器"**：给定一个 AI 产品名称，Framer AI agent 自动完成：① 收集官方来源 ② 验证 URL ③ 生成结构化分析文章 ④ 发布到站点。这与 Framer 本身的"design → publish"工作流完全对应

8. **竞品对比页面作为 CMS collection**：Framer 的 CMS collections 可以用来管理竞品数据（融资 / 定价 / 用户数），并实时展示在竞品对比页面上，与 Product-Analysis 的竞品分析逻辑一致

## 16. 今日复盘

**Framer 的核心不是"更好看的建站工具"，而是"把设计、CMS、发布、团队编辑和 AI agent 串成一个网站生产系统"**（[判断] 分析性概括，非官方口号）。

传统建站工具的核心矛盾是"设计归设计师，内容归编辑，上线归工程师"，三个角色之间存在大量沟通和等待成本（[判断]）。Framer 的创新在于：用 Canvas 作为统一编辑器，用 AI agents 作为自动化引擎，用 CMS 作为内容中枢，用 on-page editing 作为团队协作界面，让三个人变成一个人，或者至少让协作摩擦大幅降低（[判断]）。

**对 Product-Analysis 的启发** [判断 / 推断]：产品研究的核心矛盾是"分析归研究员，验证归编辑，发布归工程师"，Framer 的解题思路同样适用：用结构化的 source-first 分析框架，用 AI agent 做来源验证和内容生成，用 CMS 管理可信度评级和状态，用自动化发布取代手动同步。
