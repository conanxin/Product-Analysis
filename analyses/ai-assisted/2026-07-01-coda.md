# Coda

## 1. 一句话定位

[事实] Coda 官网首页明确将自身定位为 "Your all-in-one collaborative workspace"，并把产品文案表述为 "the collaborative, all-in-one platform that brings teams and tools together"。`coda.io/product` 的子页面 "What is Coda? The all-in-one platform for teams" 进一步把 Coda 描述为 "the collaborative all-in-one platform that blends the best of docs, spreadsheets, and applications"。`coda.io/about` 给出公司使命："To be the blinking cursor of choice for the next billion makers to collaboratively solve problems for themselves, their teams, and the world"，并指出 "80% of the Fortune 100 use Coda to improve their work"。

[事实] 维基百科 Coda (document editor) 条目把 Coda 描述为 "a cloud-based multi-user document editor"，并指出 Coda 提供 spreadsheets、presentation documents、word processor 和 apps 的能力；可能用途包括 wiki、database、project management 工具。

[判断] Coda 的产品定位不是"更强的文档"，也不是"更强的表格"，而是把文档、表格、公式、按钮、自动化和第三方应用集成（Packs）组合在同一个画布上，让一个 doc 可以承载从 wiki 到 lightweight app 的不同复杂度。

Coda 与三种产品的差异：
- **vs Notion**：Notion 主打 block + database + wiki，Coda 强调 doc-as-app 与可执行按钮。`coda.io/compare/notion` 主张 Coda "scales with your team—so you never outgrow your digital home"。
- **vs Airtable**：Airtable 是结构化记录系统，Coda 把 record 包在文档里，强调文字+表格+action 的组合。
- **vs Google Docs / Sheets / Microsoft 365**：Coda 的差异点是 doc 内可执行的 formula、button、automation，而不是文件 + 表格并列。

[事实] Coda 在 2024 年 12 月被 Grammarly 收购；Coda 官方博客（`coda.io/blog/about-coda/grammarly-acquires-coda`，2024-12-17，由 CEO Shishir Mehrotra 发布）确认收购是 "all-stock deal"，金额未披露，Mehrotra 将领导合并后的公司任 CEO。维基百科 Grammarly 条目也将 Coda 收购描述为 "all-stock deal"（金额未披露）。

[判断] 收购意味着 Coda 的产品定位正在变化：blog 描述的方向是 "the AI-native productivity suite of the future"，Coda 的 doc-as-app 能力与 Grammarly 的写作侧 AI 能力、Superhuman 的邮件侧 AI 能力有可能整合。但具体整合程度、用户数据是否合并、产品界面是否深度打通，目前公开来源不足。

[事实] P27 source-hardening 验证：Grammarly 在 2025-10-29 全面 rebrand 为 Superhuman（`grammarly.com/press` + `grammarly.com/blog/company/announcing-company-rebrand-to-superhuman` 双源验证；article:published_time 2025-10-29）。`superhuman.com/` 现为 Superhuman Suite 产品 hub，顶部导航明确列出 Mail / Grammarly / Coda / Go / Agent Store 五大产品。Coda 是该 suite 的 doc 引擎部分。

## 2. 目标用户

[事实] Coda 官网（`coda.io`、`coda.io/about`）明确点名的用户群体：
- "50,000+ teams collaborating on Coda"
- "80% of the Fortune 100 use Coda"
- 客户案例页面（如 `coda.io/solutions/case-studies/intercom`）显示大型企业 PMM、Partnerships 团队
- Blog 列出的具体客户：Zoom、Product School、Square、Intercom、Figma、Salesforce、Google 等
- Gallery 模板页展示了 50,000+ 团队使用的解决方案

[判断] Coda 的核心目标用户可以分为 9 类：

| 用户类型 | 在 Coda 中的典型用途 |
|---------|-------------------|
| **产品经理** | PRD、Roadmap、决策记录、用户研究 wiki、launch checklist |
| **运营团队** | OKR / KPI 跟踪、CRM、流程清单、跨部门协调 |
| **创业团队** | 团队 hub、轻量 CRM、内部工具、quick MVP |
| **项目管理团队** | project tracker、task management、sprint planning |
| **内部工具构建者** | 用 Packs + Buttons + Automations 拼轻量应用 |
| **文档重度用户** | wiki、knowledge base、team hub、strategy doc |
| **数据表 / workflow 用户** | tracker、inventory、CRM、approval workflow |
| **AI productivity 用户** | Coda AI 总结 / 生成 / 提取数据 / Brain |
| **Superhuman / Grammarly 生态用户** | 写作 + 邮件 + doc 的统一 AI 生产力平台 |

[判断] Coda 的用户画像与 Notion、Airtable、Asana、ClickUp 有明显重叠，但在 "doc + structured data + actions" 这条三角上比 Notion 偏 data / actions，比 Airtable 偏 doc / narrative。

## 3. 核心场景

[事实] Coda 官网首页和 `coda.io/product/docs-and-team-hubs` 把核心场景分为四类：Docs / Hubs / Trackers / Applications。

[判断] 整理后 Coda 的核心场景至少 13 个：

| 场景 | 核心承载对象 | 是否 AI 相关 |
|------|------------|------------|
| **team docs** | doc + page + table | Coda AI 总结 / 生成 |
| **project tracker** | table + view + button | Coda AI 提取 status |
| **product roadmap** | table + view + 关联表 | Coda AI 草拟 PMF |
| **OKR / planning** | table + formula + button | Coda AI 总结 OKR |
| **CRM / lightweight database** | table + form + automation | Coda AI 字段提取 |
| **meeting notes** | doc + AI column | Coda AI 提取 action items |
| **decision log** | doc + structured table | Coda AI 总结 decisions |
| **approval workflow** | button + automation + 通知 | Coda AI 起草回复 |
| **automations** | button + automation rule | Coda AI 自动响应 |
| **packs / integrations** | Snowflake / Figma / Jira / Slack / Gmail | Coda AI 跨数据源 query |
| **AI summarization / drafting** | Coda AI + Brain | Coda AI 是核心 |
| **dashboards** | table + view + chart | Coda AI 自动摘要 |
| **internal tools** | doc + pack + button + API | Coda Brain 知识整合 |

[判断] 与 Notion 的核心差别是 Coda 把 button + automation 当作一等公民，每个 doc 可以直接触发动作；与 Airtable 的差别是 Coda 把 narrative 文档当作容器，table 不是唯一对象。

## 4. 产品解决的问题

[判断] Coda 解决的是团队协作中的"文档-表格-动作"断裂问题。具体可拆为 7 个痛点：

1. **文档和数据表分离**：Google Docs / Word 写文字，Sheets / Excel 存数据，切换成本高。
2. **表格有数据但缺上下文**：Airtable 强大但缺 narrative，row 没有"为什么是这个状态"的解释。
3. **文档有上下文但缺结构化操作**：Notion 有 database 但 button / automation 弱。
4. **workflow 需要工具之间跳转**：一个简单的 approval 流程可能要在 Slack + Email + Sheets + Jira 间切换。
5. **团队知识、任务、决策、状态分散**：strategy 在 slide、task 在 Jira、knowledge 在 Confluence、decision 在 Slack。
6. **非工程团队想搭轻量应用但不想写代码**：Excel 太弱，Retool / Airtable 学习成本高，定制开发太贵。
7. **AI 需要结构化上下文才能更好工作**：纯 LLM 聊天没有 team 上下文，Coda 把团队数据 + workflow + doc 都放在一个 surface，AI 可以真正 take action。

[事实] Coda 官方 blog "Grammarly is acquiring Coda" 中 Mehrotra 明确表述："the foundational capabilities of AI change how just about every tool and surface operates"，并将 Coda 的方向定义为 "the AI-native productivity suite"。

## 6. 核心用户路径

[判断] Coda 的核心用户路径：

```
创建 doc
   ↓
添加 page / section
   ↓
插入 table
   ↓
定义 columns（text / number / select / AI column / formula）
   ↓
添加 view（Grid / Board / Calendar / Timeline / Chart）
   ↓
写 formula（可跨 doc / table / pack 引用）
   ↓
加 button（修改 row / 调 pack / 触发 automation / 弹 modal）
   ↓
设置 automation（trigger + condition + action）
   ↓
连接 pack（Slack / Gmail / Jira / Snowflake / Figma / 自建 API）
   ↓
邀请团队 + 设置权限（doc / page / row 级别）
   ↓
使用 Coda AI 总结 / 生成 / 提取 / 查询
   ↓
Coda Brain 跨数据源 query（数据来自 packs + tables + docs）
   ↓
发布模板或 internal tool 链接
```

[判断] 与 Notion 的差别：Coda 的路径多出"button → automation → pack"三步，这三步是把 doc 变成 app 的关键。

## 7. 信息架构

[判断] Coda 的核心信息架构：

| 对象 | 作用 | 所属层级 | 设计意义 |
|------|------|---------|---------|
| **Workspace** | 顶层容器，隔离数据 / 用户 | Workspace | 多团队多公司隔离 |
| **Doc** | 核心内容载体 | Doc | 一个 doc 可以包含任意 page / table |
| **Page** | 文档内分节 / 子页 | Doc > Page | 无限嵌套 |
| **Table** | 结构化数据 | Doc > Page > Table | 类似 Airtable base |
| **Row** | 表格内记录 | Table > Row | 可关联其他 table / doc |
| **Column** | 字段定义 | Table > Column | 类型丰富，含 AI column |
| **View** | 同一 table 的不同展示 | Table > View | Grid / Board / Calendar / Timeline |
| **Formula** | 跨 doc / table / pack 计算 | 任意位置 | doc-as-app 的灵魂 |
| **Button** | 触发动作 | 任意位置 | 修改 row / 调 pack / 弹 modal / 触发 automation |
| **Automation** | trigger + condition + action | Workspace / Doc | 事件驱动的 workflow |
| **Pack** | 第三方集成 | Workspace | 600+ 服务可调 |
| **Template** | 预制 doc | Gallery | 用户可发布 |
| **Permission** | 共享控制 | 多层级 | doc / folder / pack / workspace 均可控 |
| **AI (Coda AI)** | 文档内 AI 助手 | Doc / Column | 总结 / 生成 / 提取 / 翻译 |
| **Coda Brain** | 跨数据源 AI 助手 | Workspace | 知道整个团队数据 |
| **API / developers** | 编程访问 | 平台层 | 可自建 pack / 集成 |

[判断] Coda 信息架构的核心特征是"doc 是顶层容器，table / button / automation / pack 都在 doc 里"。这与 Notion 的"workspace > page > block"不同，也与 Airtable 的"workspace > base > table"不同，是第三条路。

## 8. 交互与视觉设计

[事实] Coda 官网观察到的视觉特征：
- 大色块 + 大字标题
- 卡片式布局（每个 feature / integration / customer case 一张卡）
- 客户 logo 标识带数字（"50,000+ teams"）
- 引言卡片化（Fast Company / Mark lafrate PMM at Intercom）
- CTA 按钮醒目（"Get started for free" / "Request a demo"）

[判断] Coda 交互设计的 7 个关键点：

1. **文档画布 + 结构化表格的混合**：doc 是容器，table / button / view 都在 doc 内部；用户可以同时阅读文字和操作表格。
2. **formula anywhere**：formula 不只在 cell，可以出现在 doc 任何位置，跨 doc / table / pack 引用，这是 Coda 区别于 spreadsheet 的关键。
3. **buttons as actions**：每个 button 可触发 row 操作、pack 调用、automation 触发、modal 弹出。button 让 doc 真正可操作。
4. **automations**：trigger + condition + action 模型，与 SaaS workflow 类似，但直接在 doc 里配置。
5. **packs / integrations**：600+ 服务（按官方 blog 的 800+ integrations 说法）可在 doc 内调用，本质是 no-code API wrapper。
6. **templates**：Gallery 提供 thousands of templates，是 Coda 的网络效应来源。
7. **views**：Grid / Board / Calendar / Timeline / Chart，同一份数据多种视图，对应不同角色的不同视角。

[判断] Coda 比 Google Docs 更"可操作"的原因：Google Docs 主要处理 narrative，Coda 把 formula / button / automation / pack 全部塞进 doc，让 doc 既是文档又是轻应用。

[判断] Coda 比 Notion 更偏 workflow / app-doc 的原因：Notion 的 database 偏静态（filter / sort / view），Coda 的 table + button + automation + pack 让 database 真正可执行。

[判断] Coda 强大的代价：学习成本高。新用户面对 doc + table + column + view + formula + button + automation + pack 八个层级，需要理解概念才能上手。

## 9. 内容 / 社区 / 增长机制

[事实] Coda 官网观察到的内容 / 社区资产：

- **Gallery** (`coda.io/gallery`)：thousands of templates，用户可发布
- **Blog** (`coda.io/blog`)：feature 公告 / maker stories / 客户案例
- **Help Center** (`help.coda.io`)：docs / tutorials / API reference
- **Customer cases** (`coda.io/solutions/case-studies/`)：Intercom / Zoom / Square / Figma 等
- **Compare pages** (`coda.io/compare/notion`)：针对竞品的迁移指南
- **Pack directory** (`coda.io/gallery?filter=packs`)：600+ Packs 浏览
- **Template maker community**：用户可发布自己制作的 Coda 模板到 Gallery

[判断] Coda 的增长机制依赖 5 个飞轮：

| 飞轮 | 描述 |
|------|------|
| **Templates ecosystem** | 用户做模板 → Gallery 丰富 → 新用户更易上手 |
| **Makers / internal tools community** | 内部工具 builder 创作 doc-as-app 模板 |
| **Packs marketplace** | 600+ Packs 是 Coda 的护城河，对应其他平台是 native integration |
| **PM / ops / startup teams** | 产品 / 运营 / 创业团队是高粘性用户群，自带传播 |
| **Founder-led product storytelling** | Shishir Mehrotra（前 YouTube VP）亲自发 blog / 演讲，叙事能力是 Coda 的隐性资产 |

[判断] Grammarly 收购后的新分发通道：
- Grammarly 40M DAU（按 Coda blog 数据），Coda 可作为 Grammarly 用户的高级 doc 工具
- Superhuman 邮件 + Grammarly 写作 + Coda doc = "AI-native productivity suite" 叙事
- 企业销售可以复用 Grammarly 已有 enterprise 客户群

[判断] Coda 可以从 docs 扩展到 productivity platform 的原因：

## 12. 主要优点

[判断] Coda 的 8 个主要优点：

1. **doc + table + formula 统一**：文字与数据可在同一 doc 内共存，formula 跨 doc / table / pack 引用
2. **buttons / automations 让文档可执行**：button 可触发 row 操作、pack 调用、automation、modal，doc 不再只是 narrative
3. **packs / integrations 扩展能力强**：600+ Packs（Coda blog 称 800+ integrations），是 no-code API wrapper 模式
4. **适合产品 / 运营 / 项目管理**：这三类团队的 hub / tracker / decision / OKR 都可在 Coda 完成
5. **轻量 internal tools 能力**：非工程团队可以拼出 CRM / approval / inventory 等内部工具
6. **templates 降低开始成本**：Gallery 数千 templates，新用户可快速启动
7. **AI 可利用文档和表格上下文**：Coda AI 在 doc / column 内调用，Coda Brain 跨数据源 query，AI 有真实上下文
8. **与 Superhuman / Grammarly 生态可能形成协同**：收购后 Coda 是 "AI-native productivity suite" 的 doc 引擎

## 13. 主要问题

[判断] Coda 的 8 个主要问题：

1. **学习曲线高**：doc + table + column + view + formula + button + automation + pack 八个层级，新用户容易卡住
2. **相比 Notion 心智更复杂**：Notion 的 block 模型更直觉，Coda 的 button / formula / pack 是 power-user 友好但 casual-user 困难
3. **公式 / automations 对普通用户有门槛**：formula 语法 + automation trigger 模型需要学习
4. **与 Airtable / Notion / Google / Microsoft 竞争激烈**：Coda 处于多对手交叉火力区，每个对手都有自己的优势场景
5. **收购后产品定位可能变化**：Grammarly / Superhuman rebrand 后，Coda 的独立产品路线与 suite 路线存在张力
6. **enterprise adoption 和独立品牌存在不确定性**：Coda 作为 Grammarly 子公司运营，企业用户可能担心被锁定或产品方向变化
7. **AI 功能真实采用率需验证**：官方 demo 与真实用户采用率可能有差距，Coda Brain 的实际效果需独立验证
8. **复杂 doc 容易变成维护负担**：当 doc 内有 50+ table / 100+ pack 调用 / 复杂 formula 时，文档维护成本上升

## 14. 如果我来重做

[判断] 在中文语境下做一个 Coda / doc-as-app 产品，需要回答 7 个战略问题：

**问题 1：是否应该做完整 Coda？**
不应该。完整 Coda 的 product surface 太大，中文团队不需要 doc + formula + button + pack 全套。先做垂直场景，再扩展。

**问题 2：还是先做垂直场景？**
应该。垂直场景可以是：AI 项目管理 / 内容运营 / 客户成功 / 研究协作。

**问题 3：中文团队协作痛点是什么？**
- 飞书 / 语雀 / 石墨 / 腾讯文档 / Notion 中文生态已经覆盖了大量基础场景
- 中文团队的 workflow 强依赖 IM（飞书 / 企微 / 钉钉），doc-as-app 必须深度集成 IM
- 中文团队的 data 多在 Excel / 飞书多维表格，迁移成本高
- 中文团队的 AI 需求多在"翻译 + 总结 + 中文写作"，而非跨数据源 query

**问题 4：飞书多维表格 / 语雀 / 石墨 / 腾讯文档 / Notion 中文生态 / Airtable / 低代码平台有什么特殊竞争？**
- 飞书多维表格：中文 IM 内嵌，无可替代的入口优势
- 语雀：阿里云生态，开发者向
- 石墨：早期协作文档，已有用户群
- 腾讯文档：微信生态接入，企业向
- Notion 中文：海外团队多，国内企业接受度有限
- Airtable 中文：海外团队多，localization 弱
- 低代码平台：宜搭 / 明道云 / 简道云，覆盖企业 IT 场景

**问题 5：如何和 AI agent 结合？**
中文 doc-as-app 的 AI 应该是：
- 中文 prompt 友好
- 与 IM 内的 agent 互通
- 与企业内部知识库打通
- 处理中文结构化数据（Excel / 飞书表格）有专门能力

**问题 6：如何避免成为"又一个协作文档"？**
关键是 doc-as-app 能力。文档本身没差异化，差异点是 doc 内可以搭建 workflow。

**问题 7：MVP 应该切哪个场景？**
建议：**"面向 AI 项目管理和产品研究的 doc-as-app 工作台"**

[判断] MVP 功能设计：

| 模块 | 功能 |
|------|------|
| **research doc** | 项目研究文档模板 |
| **source table** | 来源表（产品 / 文章 / 报告） |
| **claim table** | 主张 / 论据表 |
| **verification button** | 一键核对事实 |
| **review status workflow** | review 状态机（draft / reviewed / verified） |
| **report generator** | 自动生成报告草稿 |
| **agent run logs** | AI agent 执行日志 |
| **auto index sync** | 自动同步索引（README / index.yml 类比） |
| **publish dashboard** | 发布面板 |

[判断] 这个 MVP 与 Product-Analysis 本项目的需求高度吻合：当前 Product-Analysis 项目就是用 Markdown + YAML + 手动 sync 的方式管理 10+ 产品分析文章，如果用 Coda-style doc-as-app 重做，可以让 review_status / source_url_verification_status / sources 表格都成为结构化数据，自动化生成 README 和 index.yml。

## 15. 对我自己项目的启发

[判断] Coda 对 Product-Analysis 的启发是 8 条：

1. **Product-Analysis 可以把文章、sources、reports 做成多表结构**：每篇 analysis 是一个 row，文章 / sources / reports 是不同 table
2. **每个 product 是一行**：product / file / category / analysis_type / created_at / review_status / source_url_verification_status 都是字段
3. **每个 source 是 relation**：URL / type / status / used_for / note 是一行，关联到 product
4. **review_status 是 workflow state**：draft / reviewed / verified 是状态机，可由 button 触发状态迁移
5. **verification 可以由 button 触发**：点击 button 调用 validator 脚本，自动更新 status
6. **reports 是自动生成页面**：基于 article + sources 数据生成 P26 报告
7. **index.yml 类似数据库导出**：当前手工维护的 index.yml 可以由 article table 自动生成
8. **未来可做"产品研究 doc-as-app"**：把整个 Product-Analysis 仓库迁移到 doc-as-app 工作台，让研究员用 doc 而非 Markdown 写作，自动 sync 到 GitHub

- 当前仓库仍是 Markdown + YAML + Git，**短期内不应迁移**
- 中期可以在 Packs / Coda Brain 借鉴思想：sources 是 relation，verification 是 button，reports 是 view
- 长期可以用 Retool / Appsmith / Budibase 自己做 internal tool，但优先级低
